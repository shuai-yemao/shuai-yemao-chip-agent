# Ubiquitous Language — Agent System

## 系统层级

| Term | Definition | Aliases to avoid |
|------|-----------|-----------------|
| **需求对齐层** | 将用户模糊需求（自然语言 + 嵌入式术语）转化为 PRD 和 4 项施工清单的系统层 | 需求层、QA 层、分析层 |
| **Agent 编排层** | 接收施工清单，动态分配工具和任务给 Agent，用 tdd 等 skills 执行并汇总 | 调度层、执行层、Runner |
| **记忆层** | 跨层记忆系统：短期/近端/长期记忆分层存储，sqlite-vec 向量检索 | 存储层、缓存、DB 层 |
| **安全层** | 跨层权限控制、audit log、异常行为检测 | 防火墙、认证层、审计层 |
| **运维层** | 跨层监控、定期维护、健康检查 | DevOps、管理層、Ops |
| **工具层** | 贯通所有层的 skills 仓库，提供标准化工程能力 | 插件层、工具链、Library |

## 核心概念

| Term | Definition | Aliases to avoid |
|------|-----------|-----------------|
| **施工清单** | 需求对齐层的标准输出，含 4 项：现状确认/约束/施工名单/验收测试 | TODO 清单、任务列表、需求文档 |
| **PRD** | 产品需求文档，描述问题、方案、用户故事、实现决策 | 需求文档、规格书 |
| **验收测试清单** | 定义"怎么证明做对了"和"失败了怎么排查"的标准表格 | QA 用例、测试计划 |
| **ADR** | 架构决策记录，记录难以逆转的决策及其理由 | 设计文档、技术方案 |

## 角色

| Term | Definition | Aliases to avoid |
|------|-----------|-----------------|
| **用户 (User)** | 提出模糊需求的嵌入式工程师/架构师 | 客户、需求方 |
| **Chip** | 系统级架构师 AI 人格，Grill Mode 默认 | Soul、Agent |
| **执行 Agent** | 由编排层分配的临时任务执行单元，用 tdd 等技能完成工作 | Worker、Runner |

## 关系

- **需求对齐层** 产出 **施工清单** 和 **PRD**
- **施工清单** 包含 4 项：**现状确认表**、**代码约束清单**、**文件施工名单**、**验收测试清单**
- **Agent 编排层** 消费 **施工清单**，为每个任务分配 **执行 Agent**
- **执行 Agent** 使用 **工具层** 的 skills（tdd、diagnose 等）完成任务
- **记忆层**、**安全层**、**运维层** 贯通所有层
- **工具层** 贯通所有层，提供标准化能力

## Example dialogue

> **User:** "我需要加一个 I²C 驱动，支持 STM32F4 的多个设备寻址。"
>
> **Chip:** "走 Grill Flow，先对齐需求。你期望对接几个设备？速率有要求吗？"
>
> **User:** "2-3 个，400kHz 标准模式就行。"
>
> **Chip:** 需求对齐层输出
> ```
> PRD + 施工清单:
> ├─ 现状确认: 现有 GPIO I²C 软件模拟实现，无硬件 I²C 外设驱动
> ├─ 约束: STM32F4 I²C 外设限制 / 400kHz / 7-bit 寻址
> ├─ 施工名单: drivers/i2c/stm32f4_i2c.c (新) / hal_config.h (改)
> └─ 验收: 多设备读写测试 / 时序满足 400kHz / 与现有 GPIO 版本接口兼容
> ```

## Flagged ambiguities

- **"需求层"** 有两种含义：一是"需求对齐层"（我们的术语），二是"业务需求层"（偏产品）。明确用全称 **需求对齐层** 避免混淆。
- **"Agent"** 在系统中可能指 AI 执行单元（执行 Agent），也可能指编排层的调度器。明确 **编排层** 是调度者，**执行 Agent** 是被调度者。
- **"工具"** 有时指 skills（grill-me、tdd），有时指普通的 CLI 工具（gcc、cmake）。明确 **工具层** 特指 skills 仓库。
