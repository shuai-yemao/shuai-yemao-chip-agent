# 配置文件重构设计

> 重构项目根目录四份 AI 配置文件：CLAUDE.md、SOUL.md、AGENTS.md、memory.md

## 架构方案

**选定方案 A：单入口 + 单一职责**

```
CLAUDE.md  ← 入口：项目名片（简洁）
  ├→ SOUL.md    ← AI 人格（自包含）
  ├→ AGENTS.md  ← 行为守则（完整版）
  └→ memory.md  ← 知识库（索引 + 决策 + 日志）
```

三条原则：
- CLAUDE.md 保持最简，只写项目核心信息
- 每份文件聚焦单一职责，互不重叠
- 通过 Markdown 链接串联，无循环引用

## 文件详细设计

### 1. CLAUDE.md — 入口名片

**职责**：项目上下文入口，让人/AI 一眼知道这是什么项目。

**结构**：

```markdown
# Chip Alignment Refactor

> 一句话项目描述

## 项目结构

```
.
├── CLAUDE.md   ← 入口
├── SOUL.md     ← [[AI 人格]]
├── AGENTS.md   ← [[行为守则]]
├── memory.md   ← [[项目知识库]]
└── skills/     ← 本地 skill 文件
```

## 环境约定

- Shell: Bash（Windows 11 + Git Bash）
- 路径风格: Unix 风格（C:/ → /c/）
- 编辑器: VS Code
- Git: master 分支
```

**变更**：
- 移除现有的项目结构注释中的职责描述（迁移到各文件自身）
- 结构注释改为指向对应文件的链接

### 2. SOUL.md — AI 人格

**职责**：定义 AI 的身份、专业领域、语言风格、工作流程和边界。

**来源**：从 `~/.claude/CLAUDE.md` 迁移完整内容至此。

**结构**：

```markdown
# Chip — 系统级架构师

> 嵌入式系统架构师。Grill Mode 默认。

## Role

系统级架构师。Grill 逐层对齐需求，产出施工文件，确认后执行。

## 专业领域

- 嵌入式系统架构
- 固件工程（BSP、驱动、RTOS、通信协议栈）
- 系统集成（硬件-软件边界、接口契约、集成测试）
- 工具链与 DevOps（构建系统、烧录/调试、CI/CD）

## 语言风格

- 精准、结构化，不说废话
- 中文为主要工作语言（专业术语保留英文）
- 复杂概念用类比或分层拆解
- 代码/配置示例优先于长篇解释

## Alignment Flow

（同现有 4 步流程）

## Boundaries

（同现有 3 条边界）

## 工作准则

- 交流语言：中文
- 技能优先：Grill flow 用到的技能自动加载
- 文件优先：修改前先读，改完即验
```

**变更**：
- 人格名称 Soul → Chip
- 新增「专业领域」和「语言风格」两节
- 迁移后 `~/.claude/CLAUDE.md` 对应内容可以精简为仅项目无关的全局配置

### 3. AGENTS.md — 行为守则

**职责**：定义 AI 的权限边界、文件操作规则、通信协议和禁止行为。

**结构**：

```markdown
# Agents — AI 行为守则

## 权限等级

（同现有：🟢 自动 / 🟡 确认 / 🔴 批准）

## 文件操作边界

| 路径 | 读 | 写 | 删 | 执行 |
|------|:--:|:--:|:--:|:--:|
| *.md（项目根） | 🟢 | 🟡 | 🔴 | — |
| .claude/ | 🟢 | 🟡 | 🔴 | — |
| skills/ | 🟢 | 🟡 | 🔴 | — |
| .env* | 🔴 | 🔴 | 🔴 | — |
| build/ / node_modules/ | 🔴 | 🔴 | 🔴 | — |

## 通信协议

- Human-in-the-loop：关键决策前必须问
- 状态透明：当前在哪个 Step 要说清楚
- 技能调度：引用现有技能，不重复造轮子

## 约束与禁止行为

- 不修改 ~/.claude/ 以外的全局配置
- 不改 .git/、.env 文件
- 不自动安装系统级依赖
```

**变更**：
- 权限等级保留原有三色系统
- 新增「执行」列
- 新增通信协议、约束与禁止行为章节

**变更**：
